> Русская версия [CHANGELOG.md](CHANGELOG.md). При расхождениях приоритет у английской версии.

Все значимые изменения **Aspid.FastTools** документируются в этом файле.

Формат основан на [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
проект следует [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

Этот релиз сосредоточен на **инструментарии SerializeReference**: `[TypeSelector]` теперь работает с managed-reference-полями и `SerializableType`, workbench диагностики и починки находит и исправляет потерянные managed-ссылки по всему проекту, а build/CI-гейт не пропускает их в сборки.

> Если не указано иное, каждая инспекторная фича ниже работает **и в IMGUI-, и в UIToolkit-инспекторе**.

### Добавлено

#### `[TypeSelector]` на полях `SerializableType`
- `[TypeSelector]` теперь можно вешать на поле `SerializableType` / `SerializableType<T>` (и массивы / `List<T>` из них), чтобы фильтровать его пикер типов — тот же пикер по assembly-qualified name, что и у сырого `string`-поля. Раньше drawer кидал исключение на такой форме поля.
  - Базовые типы атрибута пересекаются с generic-аргументом `T` (кандидат должен быть присваиваем обоим); `Allow` управляет тем, предлагаются ли абстрактные классы / интерфейсы.
  - `Required = true` учитывается: незаполненный `SerializableType` показывает встроенную пометку «required» и считается нарушением build/CI-гейта — в инспекторе, в сохранённых ассетах и в чисто-YAML-скане сцен.
  - Анализатор принимает `SerializableType` как валидную третью форму поля (без `AFT0001`).

#### Интерфейс `ISerializableType`
- `SerializableType` и `SerializableType<T>` теперь реализуют новый интерфейс `ISerializableType` (`BaseType` + `Type`), так что API может принимать любую из обёрток полиморфно. Editor-инструментарий (drawer-ы пикера типов и гейт обязательных полей) распознаёт поля-обёртки через интерфейс вместо сопоставления двух конкретных типов.

#### `[TypeSelector]` на полях `[SerializeReference]`
- `[TypeSelector]` теперь управляет и managed-reference-полями `[SerializeReference]` (и массивами / `List<T>` из них) — инспектор заменяет стандартный UI managed-ссылки иерархическим дропдауном выбора типа (переиспользуя `TypeSelectorWindow`).
  - Выбор конкретной реализации создаёт её экземпляр; `<None>` очищает ссылку; вложенные свойства присвоенного экземпляра рисуются инлайн под foldout; наведение на дропдаун показывает полную идентичность `Namespace.Class, Assembly`.
  - Набор кандидатов по умолчанию — объявленный тип поля; базовые типы (например, `[TypeSelector(typeof(IMelee))]`) сужают его дальше.
- **Открытые generic-реализации** (например, `Modifier<T>`) тоже предлагаются. Когда аргументы выводимы из closed-generic-поля, закрытый тип создаётся сразу; иначе вторая страница пикера собирает каждый аргумент (учитывая его constraints) и валидирует результат перед созданием экземпляра.
- **Перенос данных при смене типа** — поля, общие для старой и новой реализации (по имени и сериализованной форме), сохраняют значения вместо сброса.
- **Copy/Paste** в контекстном меню заголовка поля: копирует значение managed-ссылки и вставляет его как независимый экземпляр в любое совместимое поле (вставка недоступна, если тип в буфере не присваиваем).
- **Флаг `Required`** у `[TypeSelector]` для обеих форм поля: незаполненная обязательная managed-ссылка (null) или type-name-поле (пустая строка) показывает встроенное предупреждение в инспекторе и считается нарушением build/CI-гейта.
- **Мультиредактирование объектов**: смешанное выделение показывает mixed-состояние дропдауна; выбор типа (или вставка) применяет независимый экземпляр к каждому выделенному объекту одной Undo-группой. Пер-ассетные уведомления (missing / shared) при мультивыделении скрываются.
- **Дедупликация при дублировании**: дублирование элемента `[SerializeReference]`-списка (Duplicate / Ctrl+D / добавление через `+`) больше не создаёт алиас ссылки — копия молча становится независимым экземпляром одним шагом Undo. Существовавшие ранее алиасы не трогаются.
- **Уведомления о разделяемых ссылках**: поля, стоящие за одним экземпляром, помечаются компактным уведомлением, чьё действие **Make unique** (также ПКМ → **Make Unique Reference**) клонирует независимый экземпляр.
  - Каждая shared-группа получает детерминированные цветную полоску, чип и бейдж (`Shared reference #1`, `#2`, … — id группы, а не число участников); тултип уведомления перечисляет остальных участников по пути в инспекторе (`Sidearms › Element 1`), а клик прокручивает к следующему участнику группы и коротко подсвечивает всех участников цветом группы. Всегда включено. (Окно **Managed References** раскрашивает свои per-rid-чипы независимо, так что две поверхности не обязательно совпадают цветом.)
- **Удобства авторинга**: перетаскивание MonoScript на поле присваивает его тип; долговременные именованные шаблоны (`Save as Template…` / `Paste Template ▸`), сохраняющие настроенный экземпляр на проект; **Link to Existing** — сознательно разделить один экземпляр между полями одного объекта (обратное к Make unique); `+` списка с пикером, открывающий выбор типа вместо дублирования последнего элемента; и **Create New Script…**, генерирующий заготовку подкласса типа поля и присваивающий её после компиляции. (UIToolkit получает `+` списка автоматически; IMGUI-инспектор подключается per-list через `SerializeReferenceIMGUIList.Draw`.)

#### Обнаружение и починка потерянных ссылок
- **Инлайн-починка**: потерянный тип managed-ссылки показывает компактное янтарное уведомление, чей подчёркнутый **Fix** открывает пикер типов; выбранный тип перенаправляет ссылку, сохраняя её данные. Наведение на уведомление показывает полную деталь потерянного типа.
  - Обнаружение читает осиротевший тип прямо из YAML ассета — Unity не отдаёт потерянный тип через API сериализации и не хранит его на живом объекте для префабов/GameObject (UUM-129100).
  - Сохранённые ассеты (ScriptableObject, префаб-ассеты) переписываются в их YAML; объект, открытый в Prefab Mode, чинится на живом экземпляре (его открытая stage затёрла бы правку файла при сохранении) — восстанавливая данные поля, которые Unity ещё держит, и очищая ставшие осиротевшими вложенные ссылки, чтобы баннер про missing types погас.
  - Починка находит ссылку на любой глубине — через вложенные managed-ссылки и обычные `[Serializable]`-контейнеры (поля структур/классов, элементы `List<T>`).
  - Заголовок сломанного дропдауна тонируется тем же предупреждающим янтарным, обрезается **слева**, чтобы имя класса из `<Missing Namespace.Class>` оставалось видимым, а наведение показывает полную сохранённую идентичность вместе со сборкой.
- **Smart Fix**: уведомление ранжирует наиболее вероятную замену и предлагает её вторым кликабельным сегментом (`· → Pistol?`). Ранжирование: объявленное переименование `[MovedFrom]` (высший приоритет), то же простое имя в другом пространстве имён / сборке, переименование только регистра, близкие имена, поднятые совпадением формы сериализованных полей. Берётся только из типов, которые предложил бы пикер; никогда не применяется автоматически.
- **Переименования `[MovedFrom]` отличаются от настоящих поломок.** Сохранённый тип, который больше не загружается, но заявлен ровно одним типом через объявленный `[MovedFrom]` (неоднозначная заявка не разрешается ни во что), — не поломка: Unity мигрирует его в памяти при загрузке; старое имя остаётся только в YAML.
  - Такие записи рисуются спокойными info-тонированными **pending migrations**, а не янтарной тревогой — на каждой поверхности: тост о поломке называет их автомигрируемыми, групповая карточка Project References предлагает авторитетный однокликовый **Migrate all (N) → NewType** (тот же поток confirm + diff-превью + Undo), а граф Asset References показывает info-пилюлю, нейтральную полосу `Fix ▼`, однокликовую строку **Migrate → Type** и отдельный счётчик в заголовке.
  - Build/CI-гейт согласен: pending migration никогда не считается нарушением. Запекание переименования в файлы — то, что позволит в итоге удалить атрибут `[MovedFrom]`.
  - По-настоящему потерянная карточка показывает то же однокликовое **Smart Fix**-предложение (`→ Name?`) на каждой missing-type-поверхности.
- **Проактивное обнаружение поломок**: после переименования/удаления скрипта или реимпорта ассета ссылки, которые *только что* стали потерянными, поднимают одно уведомление (`N managed references became missing — open Repair`) с deep-link в окно Repair и предложениями, ранжированными Smart Fix. Строго наблюдательное — существовавшие ранее поломки не тревожат заново, ничего не применяется автоматически; тумблер в Project Settings его выключает, а повторное включение молча пере-базлайнится, так что существующая потеря никогда не тревожит.
- **Страж удаления**: удаление C#-скрипта, используемого как managed-ссылка где-либо в проекте, сначала предупреждает — с числом использований и примерами затронутых ассетов — и даёт отменить. Unity делает это для компонентов, но никогда — для managed-ссылок.
- **Diff-превью**: подтверждение массового `Fix all` показывает точные YAML-строки, которые изменятся (старое → новое), перед необратимой перезаписью.
- **Закрытые пробелы починки**: потерянные типы на объектах в **сохранённых сценах** обнаруживаются и чинятся в памяти (через `GlobalObjectId`); окно Managed References предлагает **Open Source Prefab**, чтобы спуститься в исходник вложенного префаб-экземпляра, где живут его данные; осиротевшие записи `RefIds` (на них не указывает ни одно поле) получили кнопку **Clear** на запись.

#### Окна workbench SerializeReference
- Окно **Repair Missing References**: сканирует YAML выбранного префаба/ScriptableObject и перечисляет *каждую* осиротевшую managed-ссылку — на любой глубине вложенности и на любом дочернем объекте — с пикером **Fix** у каждой. Достаёт ссылки, которые per-field drawer показать не может (компоненты на дочерних объектах вне Prefab Mode, массовая починка, осиротевшие записи), и никогда не требует Prefab Mode.
  - **Режим всего проекта**: `Scan Project` проходит каждый `.prefab` / `.asset` / `.unity` под `Assets/`, группирует сломанные ссылки по сохранённому типу и предлагает `Fix all (N)` на группу — один выбор типа плюс подтверждение перенаправляет каждую запись во всех затронутых файлах (прямое редактирование файла; Undo той же сессии на сводке откатывает его) — плюс однокликовый Smart-Fix-quick-apply на группу. Записи, открытые в сцене или Prefab Mode, при массовом применении пропускаются (закройте и пересканируйте, чтобы включить их).
- Окно **Managed References**: строит весь граф `[SerializeReference]` ассета прямо из YAML — дерево по компонентам с корнями-указателями полей, вложенными детьми, разделяемыми (алиасными) ссылками и осиротевшими payload'ами, с бейджами `MISSING` / `SHARED`, группой `Orphaned`, детерминированными per-rid-цветами и ограниченным инлайн-пикером **Fix** для потерянных записей. Только чтение, кроме этого Fix; показывает ссылки на любой глубине и орфаны, до которых инспектор добраться не может.
  - Вложенные ссылки подписаны полным путём поля (`_primaryWeapon._chargeEffect`); неприсвоенные `[SerializeReference]`-поля показываются тусклыми слотами `<None>`, так что очищенная или никогда не заданная ссылка остаётся видимой.
  - Активная вкладка и инспектируемый ассет переживают переключение вкладок и domain reload.
- **Индекс использований и Find Usages**: проектный индекс использований managed-ссылок (строится инкрементально при импорте, по образцу индекса Id-системы) сопоставляет каждый сохранённый `[SerializeReference]`-тип с ассетами, документами и rid, где он используется. Питает `Find Usages of <Type>` в контекстном меню поля и Quick-Search-провайдер `sr:` (`sr:IWeapon`), перечисляющий все места использования с пингом их ассета, — и делает повторные запуски `Scan Project` почти мгновенными.
- **Build / CI-гейт**: `IPreprocessBuildWithReport` предупреждает или валит сборку плеера при потерянных managed-ссылках, а headless-вход `-executeMethod Aspid.FastTools.SerializeReferences.Editors.SerializeReferenceCiGate.RunCheck` сканирует проект, пишет отчёт и выходит с ненулевым кодом для CI.
  - Строгость (`Off` / `Warn` / `Fail`, по умолчанию Warn) хранится в коммитящемся `ProjectSettings/SerializeReferenceSharedSettings.asset` — не в пер-машинных `EditorPrefs` — и потому доезжает до чистого CI-раннера: `Off` пропускает, `Warn` логирует, но выходит с 0, `Fail` выходит с 1 при нарушениях.
  - Проверка `-srGateRequired` покрывает префабы, ScriptableObject **и сцены** — `.unity` читается чисто-YAML-проходом (required-поля каждого MonoBehaviour разрешаются по guid его `m_Script`), поскольку объекты сцены нельзя загрузить для инспекции.
  - Флаги `-srGateReport <path>`, `-srGateWarnOnly` (принудительный код 0) и `-srGateFail` (принудительный код 1 при нарушениях) переопределяют закоммиченную строгость на один запуск.

#### UX пикера типов
- **Атрибут `[TypeSelectorDisplay]`** (`Aspid.FastTools.Types`, editor-only `[Conditional("UNITY_EDITOR")]`) для настройки того, как тип выглядит в пикере: `Name` — переопределение отображения для строк и заголовка закрытого дропдауна (поиск по-прежнему матчит реальное имя типа; hover-тултип сохраняет реальную идентичность `Namespace.Class, Assembly`), `Group` — путь (например, `"Combat/Melee"`), помещающий тип в явную иерархию **вместо** его пространства имён (сегменты пути разделяются между типами, так что связанные типы встречаются под одним узлом), `Tooltip` на строке типа и `Icon` (имя из `EditorGUIUtility.IconContent`, путь к ассету относительно проекта с расширением или путь `Resources`-текстуры без него).
- **Секции Favorites и Recent** на корневой странице пикера: появляющийся при наведении переключатель ★ закрепляет тип в Favorites; последние выбранные типы (5 по умолчанию, настраивается) держат MRU-порядок в Recent. Обе секции сохраняются на проект в `EditorPrefs`, чистятся от типов, которые больше не разрешаются, показывают только типы из текущего набора кандидатов и скрываются при поиске.
- **Более насыщенные строки**: заголовки Favorites/Recent и строки пространств имён показывают тусклый выровненный вправо счётчик содержащихся типов (для пространств имён — рекурсивный, видимый в свёрнутом виде); текущее значение поля носит зелёную ✓ и жирный заголовок (`<None>` получает их при пустом поле), так что сохранённое значение читается отдельно от клавиатурной подсветки; разделитель отделяет закреплённый блок (`<None>`, Favorites, Recent) от иерархии пространств имён.
- `TypeSelectorWindow.Show` получил необязательный параметр `TypeSelectorFilter`: его `Predicate` дополнительно сужает список кандидатов после проверок базового типа и `TypeAllow` (drawer SerializeReference использует его, чтобы исключить `UnityEngine.Object`, строки и делегаты), а `AdditionalTypes` добавляет записи, которые скан присваиваемости найти не может (например, открытые generic-определения).

#### Настройки и Preferences
- **Вкладка Settings — секция Type Selector**: тумблер, скрывающий секцию Favorites корневой страницы (сохранённый список переживает и возвращается), слайдер ёмкости Recent (0–20, по умолчанию 5; 0 работает и как выключатель — прячет секцию и приостанавливает запись, не стирая историю) и строка обслуживания сохранённых списков, очищающая Favorites / Recent за подтверждением с указанием количества.
- **Вкладка Settings — секция Welcome**: тумблер **Auto-show Welcome**, подавляющий автооткрытие вкладки Welcome (пункт меню продолжает работать); покрывается пер-пользовательским сбросом (auto-show → on).
- **Полоски области хранения**: поскольку вкладка теперь смешивает командные и личные настройки, каждая строка помечена зелёным (коммитящийся `ProjectSettings`-ассет) или синим (пер-пользовательские `EditorPrefs`), расшифровка — компактной легендой; футер, закреплённый под скроллом, предлагает **Reset to defaults** отдельно на каждую область (**Shared** / **Per-user**), каждая за подтверждением, называющим точные восстанавливаемые умолчания. Сохранённые списки Favorites / Recent — данные, а не настройки, и переживают сброс.
- **Preferences → Aspid FastTools** (раньше только тема) теперь полное зеркало вкладки Settings окна — те же секции References / Type Selector / Welcome, легенда областей и футер per-scope-сброса, отрисованные на брендированной поверхности. Каждая поверхность строится из одного определения контролов на область и зеркалит остальные вживую; страница SerializeReference в Project Settings по-прежнему показывает контролы References в нативном виде Unity.
- Страница **Project Settings → Aspid FastTools → SerializeReference**: авто-дедупликация алиасов, обнаружение поломок, строгость build/CI-гейта и исключённые из скана папки. Обнаружение поломок — пер-машинное (`EditorPrefs`); остальное должно вести себя одинаково у каждого участника команды и в CI, поэтому живёт в коммитящемся `ProjectSettings/SerializeReferenceSharedSettings.asset`.

#### Диагностики анализатора
Поставляются в собранной Roslyn-DLL `Aspid.FastTools.Analyzers`:
- `AFT0004` (error) — `[SerializeReference]` + `[TypeSelector]` на типе, наследующем `UnityEngine.Object`, который Unity не сериализует как managed-ссылку.
- `AFT0005` (warning) — ни один видимый конкретный Unity-сериализуемый тип не удовлетворяет одновременно базе `typeof(...)` и типу элемента поля, так что пикер был бы пуст.

#### Примеры (Samples)
- Новый устанавливаемый пример **SerializeReferences** (импорт через Package Manager), демонстрирующий managed-reference-пикер на одиночных полях, `List<T>`, абстрактных базах, сужении, вложенных ссылках, generics и `Required`, в обоих инспекторах, с пошаговым `TUTORIAL` и обучающей сценой `TypeSelectorTutorial`.
  - Включает заранее сломанные ассеты, покрывающие пути восстановления: типы `Ghost*` без правдоподобного преемника — ручной пикер **Fix** (инлайн, весь ассет и проектный проход); `MovedWeaponPreset.asset` — хранит `Pistol` под старым пространством имён, как после переноса без `[MovedFrom]`, — показывает однокликовый **Smart Fix** (`→ Pistol?`); `RenamedWeaponPreset.asset` — хранит старое имя класса переименованного через `[MovedFrom]` `Crossbow` — показывает поток миграции **Migrate all**.

### Изменено
- Вкладка Welcome автооткрывается раз на **версию** пакета, а не раз на проект — после установки *или обновления* пакета следующий запуск редактора покажет её снова. Тумблер **Auto-show Welcome** по-прежнему подавляет любое автооткрытие; пункт меню не затронут.
- Окна **Repair Missing References** и **Managed References** объединены в единый workbench, чьи вкладки — отдельные пункты меню под `Tools → Aspid 🐍 → FastTools`: **Welcome**, **Asset References** (граф ссылок с инлайн Fix / Clear / Open Source Prefab — поглотил старый пер-ассетный список починки), **Project References** (проектная групповая массовая починка) и **Settings**. Результат в Project References ведёт прямо в граф Asset References этого ассета.
- Пер-свойственная проба missing-типа теперь кэширует YAML ассета по пути + времени записи, устраняя `File.ReadAllLines` на каждый IMGUI-repaint.
- Диалог подтверждения очистки потерянной managed-ссылки теперь называет, сколько полей он обнулит, — алиасная ссылка, разделённая между несколькими слотами, делает свой сброс всех указателей явным перед необратимой правкой YAML (очистка по-прежнему обнуляет все алиасные поля — изменилась только формулировка).
- `[TypeSelector].Allow` теперь по умолчанию `TypeAllow.All` вместо `TypeAllow.None`: type-name-поле (`string` / `string[]` / `SerializableType`) теперь по умолчанию показывает в пикере абстрактные классы и интерфейсы — передайте `Allow = TypeAllow.None`, чтобы ограничить его конкретными типами. Это выравнивает пикер сырого `string` с `SerializableType`, который их уже предлагал. Managed-ссылка `[SerializeReference]` не затронута: этот путь никогда не предлагал абстрактные/интерфейсные типы и полностью игнорирует `Allow`.
- `ComponentTypeSelector` теперь скрывает встроенную строку **Script** в Inspector, пока присутствует селектор — сменой типа управляет его дропдаун, поэтому избыточная строка (позволявшая менять скрипт вручную в обход селектора) убрана. Только UIToolkit-инспекторы; устаревший IMGUI-инспектор рисует эту строку сам и не затронут.

### Исправлено
- `null`-элемент в `Type[]`-члене, на который ссылается `string` / `SerializableType` `[TypeSelector]`-поле, больше не кидает `NullReferenceException` и не обрывает пикер типов — null-записи отфильтровываются до построения списка кандидатов. ([#51])
- Редактор managed-ссылок в YAML ассетов (`SerializeReferenceYamlEditor`) укреплён против не-Unity / странно отступленных файлов: измеритель отступа теперь считает табы, оставаясь согласованным с ограничивающими запись `- rid:` регулярными выражениями, а деструктивные записи (перезапись типа, удаление записи, обнуление ссылки) отказываются от файла без директивы `%TAG !u!` и выходят при неожиданных табах / смешанных отступах, чтобы не рисковать неверно ограниченной, неоткатываемой правкой. Unity всегда пишет YAML с пробельными отступами и преамбулой-директивой, так что корректные ассеты не затронуты.
- UIToolkit- и IMGUI-drawer'ы managed-ссылок больше не расходятся в размещении уведомлений: missing / required / mixed-values рисуются только при пустом поле (их позиция читается одинаково в обоих рендерерах), а shared-уведомление намеренно стоит в самом низу — под вложенными свойствами присвоенного экземпляра — в обоих рендерерах.
- **Make unique**, тихая дедупликация в списке и смена типа с переносом данных больше не теряют вложенных `[SerializeReference]`-детей: смена типа переносит разделяемых детей по ссылке (старый родитель выбрасывается, поэтому переиспользование их и сохраняет), а потоки Make-unique глубоко копируют весь граф ссылок — внутренняя топология алиасов сохранена, циклические графы безопасны.
- Циклические графы managed-ссылок больше не вешают редактор или CI-запуск: покадровый обход алиасов, скан кандидатов Link-to-Existing и обход required-полей CI-гейта не заходят повторно в экземпляр, уже находящийся на пути обхода, а Link to Existing больше не предлагает алиасного предка — именно тот выбор, что мог замкнуть поле в самоцикл.
- Quick-Search-провайдер `sr:` теперь только явный, так что набор текста в обычном окне Search больше не прогревает холодный индекс использований за модальным полнопроектным проходом. Индекс и проходы стража удаления — чистые текстовые сканы (больше не загружают каждый ассет), проваленный прогрев сбрасывает индекс в холодное состояние вместо выдачи частичных ответов всю сессию, а удаление **папки** со скриптами, на которые есть ссылки, теперь предупреждает так же, как одиночный скрипт.
- Бухгалтерия shared-ссылок стала честной: пустые поля больше не образуют фантомные «shared-группы» (в нумерации бейджей были дыры); IMGUI-пути Make-unique / Smart-Fix сбрасывают покадровый мемо алиасов, так что repaint того же кадра не показывает устаревшее уведомление; undo/redo инвалидирует этот мемо один раз глобально, а не на каждое живое поле; проба missing-типа мемоизируется на repaint (легитимно пустые поля платили полный YAML + разрешение позиции на каждый вызов); клик по shared-уведомлению прокручивает собственный скроллер инспектора (участники списков никогда не попадали в кадр); кэш ранжирования Smart-Fix очищается при внешнем изменении ассетов (например, `git checkout`).
- Граф Asset References отказывается переписывать YAML, пока ассет открыт в сцене или Prefab Mode — следующее сохранение открытой копии молча затёрло бы починку (Project-вью это уже охранял). Undo-квитанция Project References откатывает только записи, всё ещё держащие ровно тот тип, что она применяла (старая квитанция больше не может уничтожить более новую починку), и сообщает фактически затронутое число; массовое diff-превью считает только вычислимые записи.
- Прочая корректность: `SerializableType.Type` больше не кидает для созданного из кода экземпляра (null-имя в хранилище); required-уведомление string-поля больше не захватывает disposable `SerializedProperty` в отложенных колбэках; Recent записывает открытое определение сконструированного generic (закрытая форма никогда не могла показаться и молча вытесняла настоящие записи); починки в Prefab Mode выходят при грязной stage (реплей sibling-индексов мог попасть не в тот объект ассета); страж дубликатов срабатывает только на точный рост на один элемент, так что массовое восстановление (Paste Component Values, Revert) больше не дедуплицирует намеренное разделение молча; зеркала настроек переживают перестыковку окна и клик по переключателю (страж фокуса теперь пропускает только незавершённые правки текста); переопределение темы хранится на проект; нейтральные цвета `AspidSwitch` переключаются со скином редактора, чтобы ручка была видна на светлой теме; строка `<None>` больше не носит ✓ текущего значения в пикерах, где текущего значения нет вовсе (добавление `+` в списке, Fix потерянного типа, массовый проектный пикер).

### Добавлено (внутреннее)

### Добавлено

#### ProfilerMarkers
- Метод-расширение `this.Marker()`, разрешающийся в `ProfilerMarker`, уникальный для call-site (объемлющий тип + метод/поле/свойство + номер строки).

#### Serializable Type System
- `SerializableType` — `[Serializable]`-обёртка над `System.Type`, хранящая assembly-qualified name и лениво разрешающая тип при первом обращении; неявное преобразование в `Type`.
- `SerializableType<T>` — generic-вариант с ограничением базового типа, применяемым и на этапе компиляции, и в editor-пикере.
- `TypeSelectorAttribute` — `PropertyAttribute` (editor-only через `[Conditional("UNITY_EDITOR")]`), включающий пикер типов на `string`-полях и позволяющий ограничить пикер одним или несколькими базовыми типами.
- `TypeAllow` — `[Flags]`-enum, подключающий к пикеру абстрактные классы (`Abstract`), интерфейсы (`Interface`) или и то и другое (`All`); по умолчанию только конкретные классы.
- `ComponentTypeSelector` — `[Serializable]`-помощник, показывающий в инспекторе соседний компонент типа `Component` на том же `GameObject`.
- `TypeSelectorWindow` — иерархический пикер типов на базе `EditorWindow` с деревом пространств имён, fuzzy-поиском, клавиатурной навигацией и публичным API `Show(...)` для вызова из своих редакторов.
- Property drawer'ы для `SerializableType`, `SerializableType<T>`, `ComponentTypeSelector` и `[TypeSelector]`-строк (IMGUI + UI Toolkit). UI-Toolkit-drawer рисует переиспользуемый элемент `TypeField` / `InspectorTypeField`.

#### Enum System
- `EnumValues<TValue>` — `[Serializable]` словарь с enum-ключами, переживающий сериализацию Unity и поддерживающий `[Flags]`-enum'ы.
- `EnumValue<TKey, TValue>` — одиночная запись, строительный блок словаря, доступный и отдельно.
- Кастомные property drawer'ы для обоих типов с инлайн-редактированием в инспекторе.

#### ID System (бета)

- `IdRegistryResolver` — лениво строит индекс `Type AQN → registry` при первом поиске и держит его инкрементально актуальным через `AssetPostprocessor`.

- `IdStructGenerator` (инкрементальный Roslyn source generator) эмитит struct-boilerplate (`_id`, `Id`, `__stringId`, равенство, преобразования) и поддерживает как generic-структуры-цели, так и generic-содержащие типы.
- Диагностики анализатора: `AFID001` (целевая `IId`-структура должна быть `partial`) и `AFID002` (один из генерируемых членов уже объявлен пользователем).

Общие UI-Toolkit-элементы, используемые editor-поверхностями пакета, все на базовой палитре `Aspid-FastTools-Default-Dark.uss`:

- `AspidLabel`, `AspidBox`, `AspidGradientButton`, `AspidHelpBox`, `AspidInspectorHeader`, `AspidDividingLine`, `AspidAnimatedLogo`, `AspidAnimatedTitle`, `AspidAnimatedDotsBackground`, `AspidHoverGradientOverlay`.
- USS-управляемые стилевые структуры (`AspidLabelSizeStyle`, `AspidLabelFontStyle`, `AspidDividingLineSizeStyle`, `AspidDividingLineDirectionStyle`, `AspidAnimatedLogoPulseSpeedStyle`, `AspidAnimatedLogoPulseHoverAmplitudeStyle`, `AspidAnimatedLogoLayerImageStyle`, `StatusStyle`, `ThemeStyle`, …).
- Общие помощники: `AspidStyles` (единый источник истины для имён USS-классов/свойств), `InlineStyle<T>` (помощник приоритета USS-против-кода), `DoubleClickTracker`.

#### Расширения SerializedProperty
- Fluent-цепочечные помощники в `SerializePropertyExtensions` (`SetValue`, `Apply`, `Persistent`) и partial-часть `Reflection`, открывающая backing-поле/значение за `SerializedProperty`.

#### IMGUI-скоупы
- Disposable-обёртки `VerticalScope`, `HorizontalScope`, `ScrollViewScope`, открывающие layout-`Rect` для hit-тестов.

#### Editor-помощники
- `MonoScript.GetScriptName()` и `MonoScript.GetScriptNameWithIndex()` — учитывают `[AddComponentMenu]` и добавляют индексный суффикс, когда на одном `GameObject` живёт несколько копий одного компонента.

#### Окно Welcome
- Editor-окно `WelcomeWindow` (меню `Tools/Aspid FastTools/Welcome`), перечисляющее устанавливаемые примеры пакета, разбирая `package.json`.
- `WelcomeWindowStartup` показывает окно автоматически при первом импорте.

#### Примеры (Samples)
Пять устанавливаемых примеров в `Samples~/` (конвенция UPM, импорт через Package Manager):

- `Types`, `EnumValues`, `Ids`, `ProfilerMarkers`, `VisualElements`.

#### Документация
- EN- и RU-README в корне пакета и в `Documentation/EN/` и `Documentation/RU/`, зеркалящие одно содержимое с путями к изображениям под язык.
- Пер-фичевые справочные документы рядом с каждым README: `SerializedPropertyExtensions.md`, `VisualElementExtensions.md`.
