# EnumValues&lt;TValue&gt; — пошаговый туториал

Практический тур по `EnumValues<TValue>` — сериализуемому словарю, который сопоставляет члены любого enum
значениям любого сериализуемого типа и настраивается целиком в Inspector. Проходите уроки по порядку: каждый
опирается на предыдущий и соответствует одной `STEP`-секции компонента `EnumValuesTutorial`.

**Главное правило:** объявите поле `EnumValues<TValue>`, где `TValue` — тип значения; тип enum *не* входит в
объявление поля — его вы выбираете в Inspector.

```csharp
[SerializeField] private EnumValues<float> _multipliers;   // какой enum? решается в Inspector
```

Когда enum *известен* на этапе компиляции, есть типизированный близнец — `EnumValues<TEnum, TValue>` — без
пикера типа в Inspector вовсе. Уроки 1–4 учат нетипизированному варианту; урок 5 знакомит с типизированным.

## Открыть туториал

1. Импортируйте пример: **Package Manager → Aspid.FastTools → Samples → EnumValues → Import**.
2. Откройте **`Scenes/EnumValuesTutorial.unity`** и выберите GameObject **EnumValues Tutorial**.
3. **Контрольная точка:** Inspector показывает пять нумерованных секций **STEP 1 → STEP 5**, STEP 1
   называется *"Your first mapping"*. Если поля отрисовались обычными foldout'ами — сборка примера ещё
   компилируется, дождитесь рекомпиляции. STEP 1 намеренно пуст (его строите вы), STEP 2 — наполовину готов
   (его вы заполняете), STEP 3 и STEP 5 предзаполнены, чтобы сразу изучать рабочую `[Flags]`-настройку и
   типизированный вариант.

Хотите с чистого листа? Добавьте пустой GameObject и повесьте компонент **EnumValuesTutorial**.

## Урок 1 — Первое сопоставление

**Поле:** `EnumValues<float> _step1Multipliers` — полностью пустое.

1. Кликните поле типа в шапке — откроется поисковый пикер со всеми enum, видимыми проекту. Выберите
   **`DamageType`** (наберите `damage` для фильтра).
2. Нажмите **+** на списке. Появится строка: слева **дропдаун ключа** (`Physical` — первый член), справа —
   **float-значение**.
3. Добавьте ещё несколько строк с разными ключами и значениями, например `Physical = 1.0`, `Fire = 1.5`.
4. **Контрольная точка:** ключи строк отображаются настоящими enum-дропдаунами, а не текстовыми полями —
   drawer распознал ваш тип.

**Обратите внимание:**

- Тип enum — **обязательная** настройка: пока он не выбран, поле показывает предупреждение, а каждый поиск
  возвращает значение по умолчанию.
- Inspector позволяет дублировать ключи, но поиск всегда выигрывает **первая** строка с данным ключом —
  держите ключи уникальными.
- Сторона значения — обычное сериализуемое поле: здесь `float`, но подойдёт любой сериализуемый тип
  (`Color`, `Sprite`, свой `[Serializable]`-класс…).

## Урок 2 — Populate Missing Enum Members и значение по умолчанию

**Поле:** `EnumValues<Color> _step2Colors` — тип enum уже выбран (`DamageType`), список пуст.

1. **Кликните правой кнопкой** в любом месте поля и выберите **Populate Missing Enum Members**.
2. Появятся четыре строки — по одной на каждый член `DamageType` — каждая заполнена **Default Value** (белым).
3. Перекрасьте пару строк: `Fire` — оранжевый, `Ice` — голубой. Остальные оставьте или удалите ненужные.
4. Кликните правой кнопкой ещё раз — пункт меню теперь **неактивен**: недостающих членов нет.

**Обратите внимание:**

- **Default Value** (последнее поле) играет две роли: затравка для добавляемых строк и fallback, который
  `GetValue` возвращает, когда ни одна запись не совпала. Строка на каждый член **не обязательна** —
  сопоставьте исключения, остальное покроет default.
- Populate не трогает существующие строки — только дописывает недостающие, так что он безопасен на
  полузаполненном списке.
- Удаление и перестановка строк работают как в любом списке Unity; порядок важен только для `[Flags]`-поиска
  (следующий урок).

## Урок 3 — `[Flags]`-ключи и правила поиска

**Поле:** `EnumValues<float> _step3SpeedByStatus` — предзаполнено, ключ — `[Flags]` enum `StatusEffect`.

Поскольку `StatusEffect` — `[Flags]`, каждый ключ отображается **мультиселект-дропдауном флагов**, а
комбинация вроде `Burning | Slowed` — обычная самостоятельная запись. Предзаполненный список:

| # | Ключ | Значение |
|---|---|---|
| 0 | `Burning` | `0.9` |
| 1 | `Frozen` | `0.2` |
| 2 | `Slowed` | `0.5` |
| 3 | `Burning \| Slowed` | `0.4` |

`GetValue` разрешает `[Flags]`-поиск в три шага:

1. **Точное совпадение выигрывает первым, независимо от порядка** — композитная запись `Burning | Slowed`
   намеренно стоит *последней*, но поиск ровно `Burning | Slowed` возвращает `0.4`.
2. **Нет точного совпадения → выигрывает первая содержащаяся запись** — у поиска `Burning | Frozen | Slowed`
   точной записи нет, поэтому побеждает первая по порядку строка, все флаги которой содержатся в значении:
   `Burning`, `0.9`.
3. **Ничего не совпало → Default Value** — у `Stunned` записи нет: `1.0`. `None` (ноль) совпадает только с
   записью `None` и никогда — с флаговой.

Все три правила вы проверите в следующем уроке.

## Урок 4 — Чтение значений из кода

**Поля:** `DamageType _step4DamageType`, `StatusEffect _step4Effects` — ключи поиска.

Весь API — один вызов:

```csharp
float multiplier = _step1Multipliers.GetValue(_step4DamageType);
float speed      = _step3SpeedByStatus.GetValue(_step4Effects);
```

**Попробуйте** (Play Mode не нужен — работает в Edit Mode):

1. Кликните правой кнопкой по заголовку компонента и выберите **Log Tutorial Lookups**. Console напечатает по
   строке на каждый поиск плюс итерацию по записям STEP 3.
2. Поставьте `_step4Effects` в `Burning | Frozen | Slowed` и залогируйте снова → `0.90` (правило 2: первая
   содержащаяся запись).
3. Поставьте `Stunned`, затем `None` → оба раза `1.00` (правило 3: значение по умолчанию).

**Обратите внимание:**

- `GetValue` принимает любое значение настроенного enum — регистрировать ключи в коде не нужно.
- `foreach` по `EnumValues` перебирает настроенные пары `(ключ, значение)` в порядке списка; значение по
  умолчанию в итерацию **не входит**.
- Записи разрешаются лениво при первом обращении и пере-разрешаются после правок в Inspector — ручного
  обновления нет.

## Урок 5 — Типизированный вариант: `EnumValues<TEnum, TValue>`

**Поле:** `EnumValues<DamageType, float> _step5TypedMultipliers` — предзаполнено.

Когда тип enum известен на этапе компиляции, объявите его прямо в поле вместо выбора в Inspector:

```csharp
[SerializeField] private EnumValues<DamageType, float> _multipliers;   // enum зафиксирован при компиляции

float m = _multipliers.GetValue(DamageType.Fire);   // принимает DamageType, а не упакованный Enum
```

1. Выберите GameObject туториала и разверните **STEP 5**. **Контрольная точка:** в шапке **нет строки выбора
   типа** — строки сразу отображаются типизированными дропдаунами `DamageType`.
2. Правый клик по полю → **Populate Missing Enum Members** работает ровно как в уроке 2 (здесь не хватает
   только `Poison`).
3. Запустите **Log Tutorial Lookups** — строка STEP 5 использует тот же ключ `_step4DamageType`, что и STEP 1,
   так что оба варианта можно сравнить бок о бок.

**Обратите внимание:**

- Всё из уроков 1–4 переносится: те же строки в Inspector, *Populate Missing Enum Members*, fallback на
  Default Value, правила `[Flags]`-поиска, `foreach` по настроенным записям.
- `GetValue` безопасен на этапе компиляции — он принимает `DamageType`, поэтому искать по ключу чужого enum
  просто не получится.
- Сериализованный layout совпадает с `EnumValues<TValue>`: переключение поля между двумя вариантами (с тем же
  настроенным enum) сохраняет записи. `DamageDealer._damageColors` в этом примере прошёл ровно такую миграцию.
- **Правило выбора:** enum известен заранее → `EnumValues<TEnum, TValue>`; enum выбирается на стороне данных
  в Inspector → `EnumValues<TValue>`.

## Когда enum меняется

В реальных проектах члены enum переименовывают и удаляют. Ключи хранятся **строками имён**, поэтому:

- **Переименованный / удалённый член** — сохранённый ключ больше не парсится. Inspector мигрирует строку на
  первый член enum; в рантайме неразрешённый ключ логирует ошибку и просто никогда не совпадает (без краша).
- **Добавленный член** — ничего не ломается; у нового члена просто нет строки (→ значение по умолчанию), пока
  вы её не добавите, например через *Populate Missing Enum Members*.
- **Смена типа enum** на заполненном списке сохраняет строки и их значения и пере-парсит каждый ключ против
  нового типа — ключ, чьё имя существует в обоих типах, выживает; остальные мигрируют (и сохраняются) на
  первый член нового типа, так что при обратном переключении исходные имена ключей **не** вернутся.
  Попробуйте на STEP 1, когда закончите с ним.

> Если переименование должно сохранить данные — переименуйте сериализованные строки в файле ассета/сцены (или
> заново выберите ключ в Inspector): миграция ключа никогда не сбрасывает сторону значения.
